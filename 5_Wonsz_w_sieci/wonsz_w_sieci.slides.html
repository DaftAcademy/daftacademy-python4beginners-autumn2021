Jak się połączyć z maszyną w kambodży?¶

Adres IP (ang. IP address) liczbowy identyfikator nadawany interfejsowi sieciowemu "Interfejs (urzdzenie)"), grupie interfejsw (broadcast, multicast), bd caej sieci komputerowej w protokole IP

Uniform Resource Identifier (URI, ang. Uniform Resource Identifier, tum. Ujednolicony Identyfikator Zasobw) jest standardem internetowym umoliwiajcym atw identyfikacj zasobw w sieci

URL(Uniform Resource Locator) ujednolicony format adresowania (okrelania lokalizacji) zasobw (informacji, danych, usug) stosowany w Internecie i w sieciach lokalnych

Domain Name System¶

DNS (Domain Name System) to protok, ktrego gwna funkcja polega na tumaczeniu atwych do zapamitania przez czowieka nazw domen na zrozumiae dla komputerw dane liczbowe.

Ogólna zasada działania¶

Struktura zapytania HTTP¶

Klient wysya danie do serwera w formie wiadomoci. Wiadomo ta ma dokadnie zdefiniowany format:

  • linia okrelajca czasownik HTTP, zasb i wersj protokou,
  • linie zawierajce nagwki,
  • pust lini okrelajc koniec nagwkw,
  • ciao wiadomoci (jeli istnieje).

Struktura odpowiedzi HTTP¶

  • Linijka statusu odpowiedzi: HTTP/1.1 200 OK
  • Kolejne nagwki odpowiedzi, np.: Server: Apache
  • Pusta linia
  • Ciao (ang. body) odpowiedzi: 
    <body>  
<h1>trening</h1>  
</body>

JSON¶

  • We wspczesnych aplikacjach webowych czciej moemy spotka si z wysaniem wiadomoci do serwera w formacie JSON.
  • JSON jest specyfikowany w RFC 4627. Od momentu powstania wypiera XML ktry by najczciej uywanym formatem serializowania przesyanych danych.

Przykadowy Request logowania wysany do serwera w postaci JSONa

POST /login HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Content-Length: 39
Content-Type: application/json

{"login":"daft","password":"zexaeRL5PoeBsGYJ"}

XML¶

  • Coraz rzadziej spotkamy siz przesyaniem POSTw w formacie XML
  • Kiedy podstawowy format przesyania danych, dzisiaj spotkamy w starszych rozwizaniach enterprisowych wielkich firm napisanych najczciej w JAVA EE.
  • XML jest formatem wykorzystanym w protokole SOAP.

Przykadowy Request wysany do serwera w postaci SOAP XML

POST /login HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Content-Length: 341
Content-Type: application/soap+xml; charset=utf-8

<?xml version="1.0"?>
<soap:Envelope
xmlns:soap="http://www.w3.org/2001/12/soap-envelope"
soap:encodingStyle="http://www.w3.org/2001/12/soap-encoding">

<soap:Body xmlns:m="http://www.example.org/login">
  <m:LoginData>
    <m:Login>daft</m:Login>
    <m:Password>zexaeRL5PoeBsGYJ</m:Password>
  </m:LoginData>
</soap:Body>

</soap:Envelope>

Struktura URL¶

Scheme - Okrela protok jakigiego musimy uy aby poczy si z danym rdem \ Authority - Domena, IP z ktrym si musimy poczy \ Port - Port na jakim nasuchuje usuga \ Path to resource - cieka do zasobu na serwerze \ Parameters - Parametry \ Anchor - Dokadny fragment zasobu\

  • Kodowaniem znakw dla URI jest standard ASCI
  • Znaki ! * ' ( ) ; : @ & = + $ , / ? # [ ] sznakami zarezerwowanymi i musz by escapowane (RFC 3986)

Nagłówki HTTP¶

Nagwki wykorzystywane s do przesyania metadanych na temat zasobw. Mog zawiera na przykad informacje o formacie, statusie odpowiedzi czy dacie. Maj one posta nazwa-nagwka: warto-nagwka

Właściwości metod HTTP¶

Bezpieczna Metoda HTTP jest bezpieczna, jeli nie zmienia stanu serwera. Innymi sowy, metoda jest bezpieczna, jeli prowadzi do operacji tylko do odczytu.

Idempotentna Metoda HTTP jest idempotentna, jeli identyczne danie moe zosta wykonane raz lub kilka razy z rzdu z tym samym skutkiem, pozostawiajc serwer w tym samym stanie.

Keszowalna Metoda HTTP jest keszowalna wtedy kiedy moe zosta zkeszowana, czyli odpowied jest przechowywana w celu pobrania i wykorzystania pniej.

Statusy odpowiedzi¶

Kody stanu odpowiedzi HTTP wskazuj, czy okrelone danie HTTP zostao pomylnie zakoczone. Czyli niesie za sob informacj co si stao z naszym daniem. Odpowiedzi s pogrupowane w pi klas:

  1. Statusy Informacyjne (100199)
  2. Statusy Sukcesu :> (200299)
  3. Statusy Przekierowa (300399)
  4. Statusy Bdu Po Stronie Klienta (400499)
  5. Statusy Bdu Po Stronie Serwera (500599)

Czym się różni protokół HTTP od HTTPS?¶

Protok HTTP:

  • HTTP domylnie korzysta z portu 80.
  • Adres strony www rozpoczyna http://
  • Przesyane informacje nie s szyfrowane - loginy, hasa itp. S atwe do odczytania.

Protok HTTPS:

  • Domylny port dla HTTPS to 443.
  • Strona internetowa poprzedzona jest przedrostkiem https://
  • Poufne dane przesyane midzy klientem a serwerem s szyfrowane poprzez protok SSL / TLS.
  • Dane s zabezpieczone przed przechwyceniem, odczytaniem i modyfikacj przez osoby trzecie.

Basic Auth¶

  • Najprostszy sposb uwierzytelniania
  • Nie gwarantuje bezpieczestwa hase!
    • Uytkownik i haso s przesane jedynie zakodowane w BASE64 wic generalnie w PLAIN TEXT!
  • Bezpieczestwo (czciowe) stanowi jedynie poczenie BasicAuth z HTTPS
Authorization: Basic user_name:password
 \___________/  \___/ \_______/ \______/
      |           |       |         |
    Header       typ    user      pass
             autoryzacji
  • Cao nastpnie jest kodowane do BASE64 (wraz z dwukropkiem)
    • user_name:password -> dXNlcl9uYW1lOnBhc3N3b3Jk
  • Nagwek za wyglda nastpujco:
    • Authorization: Basic dXNlcl9uYW1lOnBhc3N3b3Jk

Token¶

  • Podajemy username i password raz podczas logowania
  • W zamian otrzymujemy token
  • Teraz zamiast loginu i hasa moemy uywa tokenu

Cookies¶

  • Cookie to te header
  • Aktualny format i specyfikacj cookie mona znale w RFC 6265
  • Cookie powstao by przechowywa mae iloci informacji w bezstanowym protokole HTTP
  • Cookie domylnie jest przechowywane przez przegldark i zawsze doklejane do kolejnych zapyta na danej stronie lub jej podstronach (o ile parametry nie mwi inaczej)
Set-Cookie: nazwa=warto; expires=DATA; path=CIEKA; domain=DOMENA; secure
 \________/ \____/ \_____/          \__/       \_____/         \____/  \____/
      |       |       |              |            |               |       |
   Header   nazwa   warto  data wyganicia/   path      ograniczenie  HTTPS 
           cookie   cookie   czas ycia cookie                domeny     only
  • Budowa Cookie
    • nazwa cookie - Konkretna nazwa pliku do ktrego zostanie wpisana warto w ASCI bez znakw specjalnych
    • warto - Dowolna warto w ASCI bez znakw specjalnych (dlatego czsto kodowane w BASE64)
    • expires - Cookie moe wygasa po zadanym czasie w sekundach, w okrelonej dacie lub nigdy
    • path - cieka dla ktrej cookie bdzie dostpne
    • domain - Ograniczenie domenowe np do konkretnej subdomeny danej strony
    • secure - Nie przyjmuje wartoci, okrela ze cookie moe by wykorzystane tylko HTTPS
  • Cookie moe przetrzymywa sesj uytkownika (domylne dla wielu frameworkw w python np.: Flask)
  • Cookie to te nagwek !! a nagwki maj swoj maksymaln dugo ograniczajc wielo sesji uytkownika
  • Cookie moe by dowolnie usuwane przez przegldark czy innego klienta, ale nie modyfikowane. ( w teorii)

Jak zbudowane są strony internetowe?¶

MPA (Multi Page Application) - Strona internetowa ktra renderowana jest po stronie serwera. W odpowiedzi na zapytanie dostajemy gotowego HTMLa. Poruszajc si po stronie jest ona przeadowywana.

SPA (Single Page Application) - Aplikacja napisana w JavaSciptcie dziaajca w przegldarce i renderowana bezporednio w przegldarce. Podczas poruszania si po stronie nie przeadowujemy aplikacji. Zazwyczaj komunikuje si z serwerem (backendem) aby pobra lub zapisa dane.

REST API¶

Interfejs API, czyli aplikacyjny interfejs programistyczny, jest zestawem regu definiujcych sposb, w jaki aplikacje lub urzdzenia mog si ze sob czy i komunikowa. Interfejs API REST to interfejs API zgodny z zasadami projektowania REST, czyli stylem architektury representational state transfer. Z tego powodu interfejsy API REST s czasami nazywane interfejsami API zgodnymi ze specyfikacj REST
